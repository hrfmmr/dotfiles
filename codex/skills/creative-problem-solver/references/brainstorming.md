# Technique: Brainstorming (Diverge → Converge)

## One-liner
時間を区切ってまず発散し、壊れやすい初期案を守る。その後に明示的に収束（評価・選定・具体化）する。

## Use when
- まず幅広く案を出し、そのあと選びたい。
- 解空間が早すぎる段階で狭まっている。
- 早期評価（自己検閲含む）が強すぎる。

## Avoid when
- 1つの高シグナルな再フレームだけ欲しい（Synectics / Provocation）。
- 問題の核が明確な矛盾（TRIZ）。

## Inputs
- 問題プロンプト（理想は「How might we...」形式）。
- タイムボックスと、発散/収束の明確な境界。

## Procedure (fast, 5–12 min)
1. ルール設定（30秒）: 判断を遅らせる、量を出す、乗っかる、突飛さ歓迎。
2. 発散（3〜7分）: 生案を15〜30個出す。
3. 収束（2〜5分）: クラスタ化 → 上位5件選択 → 実験として具体化。

## Procedure (full, 15–30 min)
1. Frame
   - 解ける問いへ変換する: 「How might we <成果> without <コスト> ?」
   - 制約と非目標を明示する。
2. Diverge (10–15 min)
   - 30〜60案を生成する。
   - 強制的に多様化する: 領域を交互に切る（技術/プロセス/人）、単位を変える（1リクエスト/1日/1ユーザー）。
3. Converge (5–10 min)
   - クラスタ化してテーマ名を付ける。
   - テーマごとに代表案を1つ選ぶ。
   - CPSオプション（シグナル + エスケープハッチ）に変換する。

## Prompt bank (copy/paste)
- 「Yを増やさずにXを減らす方法を20個。」
- 「予算が10分の1なら何をする？」
- 「予算が10倍なら何をする？」
- 「採用できないなら何をする？」
- 「システムの半分を消すなら何をする？」

## Outputs (feed CPS portfolio)
- 生案30〜60件。
- テーマ化されたクラスタ3〜7個。
- tierごと（Quick Win → Moonshot）に具体化した実験案。

## Aha targets
- 新しい探索軸（時間、範囲、責任、インターフェース、インセンティブ）が見える。
- 制約の捉え方が反転する（「Xできない」→「Xを安全にやる必要がある」）。

## Pitfalls & defusals
- Pitfall: ブレストが議論化する → Defusal: フェーズ境界を厳守し、批評は収束フェーズのみ。
- Pitfall: 最初の案の派生ばかりになる → Defusal: ドメインジャンプ（技術↔プロセス↔インセンティブ）を強制。
- Pitfall: ただの案リストで終わる → Defusal: 上位案を必ず測定可能な実験に書き換える。

## Examples
### Engineering
Prompt: 「信頼性を落とさずにCI時間を半分にするには？」
- 発散: 並列化、キャッシュ、テスト分割、フレーク隔離、pre-commit、プロファイリング。
- 収束: テストシャーディング + キャッシュ + フレーク隔離を選択。
シグナル: CI中央値、フレーク率。

### Mixed domain
Prompt: 「チームを疲弊させずに顧客サポート初動を速くするには？」
- 発散: トピック別ルーティング、オフィスアワー、マクロ、自己解決ドキュメント、SLA tier。
- 収束: 2週間、マクロ拡充 + トリアージキューを試す。
シグナル: 応答時間 + 満足度。
エスケープハッチ: 品質低下時はマクロ運用を戻す。
