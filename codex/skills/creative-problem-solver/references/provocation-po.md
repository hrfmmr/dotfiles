# Technique: Provocation (PO) / Lateral Thinking

## One-liner
あえて「間違った」主張（PO）を置き、それが成り立つ条件を探って、使えるレバーに変換する。

## Use when
- 常識的推論では同じ答えしか出ない。
- 最適化ではなくAha（新しい再表現）が必要。
- 実現性より、想像力がボトルネック。

## Avoid when
- 合意形成優先で心理的安全性が低い（先にSix Thinking Hats）。
- 問題設定が十分で、実行だけが必要。

## Inputs
- 問題フレーム + 制約。
- 一時的に実現性判断を保留できる姿勢。

## Procedure (fast, 6–10 min)
1. POを書く（逆転/誇張）。
2. 「POが成立するには何が真である必要があるか？」を問う。
3. 原理/レバーを2〜3個抽出する。
4. 実験（シグナル + エスケープハッチ）へ変換する。

## Procedure (full, 15–25 min)
1. Provocationを3つ作る。
   - デフォルト反転、アクター削除、手順削除、インセンティブ反転。
2. 各POについて。
   - 成立条件を列挙（「POが真なら、〜が必要」）。
   - POが露出させる隠れ制約を特定する。
3. Translation（最重要）
   - 成立条件を実行可能レバーへ変換する。
     - 自動化、計測、権限モデル変更、インセンティブ、インターフェース再設計。
4. Converge
   - 高シグナルかつ可逆なレバーを1つ選ぶ。

## Prompt bank (copy/paste)
- 「PO: 逆のデフォルトを真にする。」
- 「PO: その役割/システムを完全に消す。」
- 「PO: 全員が1分でこれを終える。」
- 「魔法なしで可能にするには何が必要か？」
- 「このPOが示す採用原則は何か？」

## Outputs (feed CPS portfolio)
- 現在の枠組みを破るレバー2〜3件。
- 言いっぱなしでない、翻訳済み実験セット。

## Aha targets
- 欠けていたプリミティブ（例: 「文章より実例」「意見よりシグナル」）の発見。
- 聖域を変数化できる。

## Pitfalls & defusals
- Pitfall: 逆張りで終わる → Defusal: 具体レバーへの翻訳を必須化する。
- Pitfall: 制約違反の提案になる → Defusal: 制約を翻訳境界として使う。
- Pitfall: テスト不能 → Defusal: 最小可逆実験へ書き換える。

## Examples
### Engineering
PO: 「誰もドキュメントを書かない。」
- 成立条件: ドキュメント自動生成、更新の暗黙同期。
- レバー: コード+実例から生成、CIで実例コンパイルを必須化。
Signal: ドキュメント鮮度改善。
Escape hatch: 重要導線のみ手書きドキュメント併用。

### Mixed domain
PO: 「会議は一切しない。」
- 成立条件: 非同期意思決定、文脈共有。
- レバー: 決定ログ + 提案文書 + コメント期限。
Signal: 意思決定が滞らない。
Escape hatch: 争点が強い論点のみ短時間同期を再導入。
