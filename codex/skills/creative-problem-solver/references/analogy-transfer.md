# Technique: Analogy Transfer (borrow a solved pattern)

## One-liner
すでに解かれている領域のパターンを借り、役割・資源・不変条件を対応付けて、対象問題に使える具体的なレバーへ落とし込む。

## Use when
- 新規性は欲しいが、実証済みメカニズムに乗せたい。
- 問題が既知の型（ルーティング、トリアージ、キュー、詐欺対策、安全系）に似ている。
- 「これは○○に似ている」という再表現から、行動に繋げたい。

## Avoid when
- 問題が純粋な組み合わせ設計（Morphological Analysis を使う）。
- 核が明確な矛盾（TRIZ を使う）。
- たとえ話が政治的・感情的にノイズになる。

## Inputs
- 対象問題（1〜2文）。
- 参照元ドメイン（1〜3個）。

## Procedure (fast, 5–10 min)
1. 解法が成熟しているドメインを選ぶ（物流、医療、セキュリティ、生態系、製造など）。
2. 借りるパターン名を決める（トリアージ、キューイング、カナリア、エスクロー、サーキットブレーカー）。
3. 次を対応付ける。
   - roles（誰/何が動くか）
   - resources（時間、注意、お金、容量）
   - signals（何を測るか）
   - failure modes（何が壊れるか）
4. 2〜3個のレバー候補に翻訳し、実験に書き換える。

## Procedure (full, 15–25 min)
1. 3つの類推候補を作る。
   - 近い領域を1つ、遠い領域を1つ含める。
2. 対応表を作る。
   - Target element → Source analog → Imported mechanism。
3. 制約を丁寧に持ち込む。
   - 「参照元の仕組みが機能する前提は何か？」を確認する。
   - それを検証すべき仮説にする。
4. 丸写しせず適応する。
   - 仕組みは残し、表面は対象に合わせる。
5. 収束する。
   - テストしやすく、可逆なレバーを優先する。

## Prompt bank (copy/paste)
- 「同じ失敗モードを、どの領域がすでに解いているか？」
- 「これがトリアージ窓口 / 工場ライン / 空港保安なら、対応する工程はどこか？」
- 「ここで最も希少な資源は何か（容量/注意/信頼）？」
- 「持ち込むメカニズムは何で、どんな前提が必要か？」

## Outputs (feed CPS portfolio)
- 新しい表現（例: これはキュー/トリアージ問題だ）。
- 持ち込み可能なメカニズム2〜5個。
- 実験候補の短いリスト。

## Aha targets
- 見る単位が切り替わる（機能→フロー、バグ→リスク露出、人→インセンティブ、要求→キュー）。

## Pitfalls & defusals
- Pitfall: 表面的なたとえ（「Uberみたいに」） → Defusal: 仕組み名（マッチング、ルーティング、価格付け）まで言語化する。
- Pitfall: 不適切な不変条件を輸入する → Defusal: 必要前提を明示し、先に検証する。
- Pitfall: 比喩だけで終わる → Defusal: 必ずレバーとシグナルに落とす。

## Examples
### Engineering
問題: サポートキュー過負荷。
類推: ERトリアージ。
- メカニズム: 重症度ルーティング、簡易案件のファストパス、エスカレーション規約。
レバー: 重症度タグ + マクロ導入。
シグナル: 初回応答時間。
エスケープハッチ: 処理時間が悪化したらタグ運用を停止。

### Mixed domain
問題: 会議でのチーム衝突。
類推: 交差点の優先通行。
- メカニズム: 明示的な順番、合図、進行役を信号機として機能させる。
レバー: 発言順 + ファシリテーションルール。
シグナル: 参加分布。
エスケープハッチ: 議論が遅くなりすぎたら構造を緩める。
